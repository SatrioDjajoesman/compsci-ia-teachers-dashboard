# Teacher Dashboard Technical Solution Document

## 1. List of Technical Approaches & Justification

This section systematically lists the core technical approaches adopted for the development of this teacher dashboard solution and justifies their applicability within the current project context.

### 1.1 Core Framework: Next.js 16 (App Router) & React 19
*   **Approach Description**: Adopting the Next.js 16 App Router architecture and utilizing React Server Components (RSC) for server-side rendering.
*   **Justification**:
    *   **Performance Optimization**: RSC allows us to fetch data directly on the server (e.g., student lists, attendance records), reducing the size of the JavaScript bundle sent to the client, thereby accelerating the Largest Contentful Paint (LCP).
    *   **Development Efficiency**: App Router provides intuitive file-system-based routing, simplifying the management of page structures and layouts.
    *   **SEO & Accessibility**: Although this is an internal dashboard, the semantic HTML generated by server-side rendering improves the application's accessibility.

### 1.2 Styling System: Tailwind CSS 4
*   **Approach Description**: Using a Utility-first CSS framework for rapid UI development.
*   **Justification**:
    *   **Consistency**: Tailwind's preset design system ensures consistency in colors, spacing, and typography, which is crucial for maintaining a professional dashboard interface.
    *   **Maintainability**: Writing styles directly in components via utility classes avoids the style conflicts and dead code accumulation associated with traditional CSS files.
    *   **Responsive Design**: The built-in breakpoint system makes it simple and efficient to adapt to different screen sizes (e.g., viewing the dashboard on a tablet).

### 1.3 State Management: Zustand
*   **Approach Description**: Using the lightweight Zustand library to manage global client-side state (e.g., currently selected class, filter criteria).
*   **Justification**:
    *   **Minimalism**: Compared to Redux, Zustand has no complex boilerplate code, making it perfect for this project's requirement to manage a small amount of global state (like `currentClass`, `selectedStudents`).
    *   **Decoupling**: It allows us to define state logic outside of components, keeping components pure and focused on UI rendering.

### 1.4 Database Interaction: Supabase (PostgreSQL)
*   **Approach Description**: Utilizing the PostgreSQL database and instant APIs provided by Supabase.
*   **Justification**:
    *   **Relational Data Integrity**: Complex associations exist between students, classes, tasks, and attendance records. A SQL database ensures strong data consistency through Foreign Keys.
    *   **Development Speed**: Supabase provides a type-safe TypeScript client (`@supabase/supabase-js`), significantly reducing the workload of writing backend API interfaces.

### 1.5 Email Service: Resend & React Email
*   **Approach Description**: Using the Resend API to send emails and React Email to build maintainable email templates.
*   **Justification**:
    *   **Developer Experience**: Building emails using React components (rather than cobbling together HTML strings) greatly improves template readability and maintainability.
    *   **Reliability**: Resend provides a high-deliverability transactional email service, ensuring parents receive notifications in a timely manner.

## 3. Product Structure & Design Principles

### 3.1 Overall Architecture
The product adopts a **Modular Monolith** architecture, where the frontend UI and backend logic are tightly integrated via Next.js **Server Actions**.

*   **UI Layer (Components)**: Responsible for displaying data and receiving user interactions. Components are divided by functional areas (e.g., `AttendanceSection`, `MailSenderSection`) to achieve high cohesion.
*   **Business Logic Layer (Actions)**: Located in the `actions/` directory, encapsulating core business rules (e.g., calculating attendance rates, email sending logic). This layer interacts directly with the database.
*   **Data Layer (Supabase)**: The PostgreSQL database stores all persistent data.

### 3.2 Module Breakdown
*   **Dashboard Module**: The core console that coordinates various sub-functions via `MainDashboard`.
*   **Attendance Module**: Handles CRUD operations for attendance records and supports batch operations.
*   **Task Module**: Manages task assignment and submission status tracking.
*   **Communication Module**: Integrates email template management and sending functions, supporting both manual and automated triggers.

### 3.3 Data Flow Design
1.  **Read Flow**: Page Load -> Server Component calls `dbactions` -> Supabase Query -> Data passed to Client Component via Props -> Zustand stores some global state.
2.  **Write Flow**: User Action -> Client Component calls Server Action -> Data Validation -> Supabase Write -> `revalidatePath` triggers page re-rendering -> UI Update.

### 3.4 Design Principle Justification
*   **Scalability**: The modular component structure makes adding new features (e.g., "Grade Management") in the future as simple as adding a new Section component and corresponding Action, without affecting existing code.
*   **Decoupling**: UI components do not contain direct database query code. All data fetching is done through the `actions` layer, making it relatively easy to swap out backend services or databases in the future.

    student_id: studentId,

## 6. Concrete & Visible Algorithms

### Core Algorithm: Student Comprehensive Performance Analytics

This algorithm is used to calculate and display student attendance rates and assignment submission status in real-time on the dashboard. It is not just simple data reading but involves data aggregation, filtering, and ratio calculation logic.

#### Algorithm Logic Description
1.  **Input**: `studentId` (Unique Student Identifier).
2.  **Data Fetching**:
    *   Execute parallel queries on `attendance_records` and `task_submissions` tables, filtering all records belonging to this `studentId`.
3.  **Task Metrics Calculation**:
    *   Iterate through submission records, counting those with status `'Submitted'` as `completedTasks`.
    *   Count those with status `'Late'` as `overdueTasks`.
4.  **Attendance Metrics Calculation**:
    *   Calculate total sessions `totalSessions`.
    *   Separately count instances of `'present'`, `'late'`, `'absent'`, and `'truant'`.
5.  **Ratio Calculation & Normalization**:
    *   If `totalSessions > 0`, calculate percentages for each status: `(Count / Total) * 100`, keeping one decimal place.
    *   If `totalSessions == 0`, default all ratios to `'0'` (handling the division-by-zero edge case).
6.  **Output**: A structured object containing raw counts and calculated ratios.

#### Pseudocode

```pseudocode
FUNCTION GetStudentAnalytics(studentId):
    // 1. Get raw data
    submissions = DB.QUERY("task_submissions").WHERE("student_id", studentId)
    attendance = DB.QUERY("attendance_records").WHERE("student_id", studentId)

    // 2. Calculate task metrics
    completedCount = COUNT(submissions WHERE status IS "Submitted")
    overdueCount = COUNT(submissions WHERE status IS "Late")

    // 3. Calculate attendance metrics
    totalSessions = LENGTH(attendance)
    presentCount = COUNT(attendance WHERE status IS "present")
    lateCount = COUNT(attendance WHERE status IS "late")
    absentCount = COUNT(attendance WHERE status IS "absent")

    // 4. Handle edge cases & calculate ratios
    IF totalSessions > 0 THEN
        presenceRate = (presentCount / totalSessions) * 100
        tardinessRate = (lateCount / totalSessions) * 100
    ELSE
        presenceRate = 0
        tardinessRate = 0
    END IF

    // 5. Return results
    RETURN {
        TaskMetrics: { completed: completedCount, overdue: overdueCount },
        AttendanceMetrics: { total: totalSessions, presenceRate: FORMAT(presenceRate, 1) ... }
    }
END FUNCTION
```

## 7. Innovation & Cleverness

### 7.1 Violation Type Driven Email Template System
**Beyond Convention**: Most systems only provide generic "Notification" functionality. This solution designs a dynamic template system based on `violation_type` (e.g., "Late attendance", "Truant attendance").
**Benefit**:
*   **Automation & Precision**: The system automatically selects the template with the most appropriate tone and wording based on the specific violation.
*   **Extensibility**: Future support for new notification scenarios only requires adding new violation types and corresponding templates to the database, without modifying code logic.

### 7.2 Hybrid State Management
**Clever Design**: We did not put all state into Zustand, nor did we rely entirely on React Context. Instead, we chose to pass **Server Data** (like student lists) via Props (leveraging Next.js caching mechanisms) while putting **UI Interaction State** (like selected items) into Zustand.
**Benefit**:
*   **Reduced Client Load**: Avoids repeated fetching of large amounts of data on the client side.
*   **Responsiveness**: UI state changes (like checking a student) are instantaneous and do not require waiting for server responses, providing a smooth user experience.

## 8. Embedded Design Choices

*   **Database Primary Key Selection**: I chose **UUID** over auto-incrementing integers as primary keys.
    *   *Reason*: This avoids ID conflicts in distributed systems or during data migration and makes passing IDs via URL parameters more secure (harder to guess).
*   **Server Actions vs. API Routes**: I chose **Server Actions** instead of traditional API Routes.
    *   *Reason*: This eliminates the boilerplate code of manually writing `fetch` requests and handling JSON serialization, making frontend and backend code look like they run in the same environment, reducing cognitive load and the potential for errors.
*   **Zod Validation (Planned)**: While currently relying mainly on TypeScript types, introducing Zod for runtime validation is a clear design direction.
    *   *Reason*: This prevents invalid form data from entering the database, improving system robustness.
